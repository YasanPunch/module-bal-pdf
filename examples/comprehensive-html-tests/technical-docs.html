1. Overview

The Ballerina HTTP client provides a type-safe, resilient way to make HTTP calls to external services. It supports automatic payload binding, configurable retry policies, circuit breaker patterns, and OAuth2/JWT authentication out of the box.

Note This guide covers the http:Client API in Ballerina Swan Lake Update 13 (2201.13.x). Some features may not be available in earlier versions. Check the changelog for version-specific differences.

Key capabilities of the HTTP client:

  • Type-safe payload binding — responses are automatically deserialized to Ballerina records
    • Supports JSON, XML, and form-urlencoded payloads
    • Validation against record types at deserialization time
    • Custom media type handlers via http:MediaTypeHandler
      • Register handlers for application/protobuf, application/avro, etc.
      • Priority-based handler resolution when multiple match
      • Built-in handlers for JSON, XML, text, and binary
  • Resilience patterns — built-in retry, circuit breaker, and failover
    • Configurable retry backoff (fixed, exponential, jitter)
    • Circuit breaker with rolling window failure counting
  • Connection pooling — automatic HTTP/1.1 and HTTP/2 connection management
  • Observability — automatic distributed tracing with OpenTelemetry and Prometheus metrics

2. Quick Start

The simplest usage creates a client and makes a GET request with automatic type binding:

import ballerina/http;
import ballerina/io;

type Album record {
    string title;
    string artist;
    decimal price;
};

public function main() returns error? {
    http:Client albumClient = check new ("https://api.example.com");

    Album album = check albumClient->/albums/["1"];
    io:println(album.title);    // "Blue Train"
    io:println(album.artist);   // "John Coltrane"
    io:println(album.price);    // 56.99
}
Tip Use resource method syntax (client->/path) instead of the older client->get("/path") style. Resource methods provide compile-time path validation and better IDE support.

3. Configuration

3.1 Client Configuration

The http:ClientConfiguration record controls client behavior. All fields are optional with sensible defaults.

Field Type Default Description
httpVersion http:HttpVersion HTTP_2_0 Protocol version. Falls back to HTTP/1.1 if server doesn't support HTTP/2.
timeout decimal 60.0 Request timeout in seconds. Applies to connection + response combined.
poolConfig http:PoolConfiguration See below Connection pool settings: max active, max idle, wait timeout.
secureSocket http:ClientSecureSocket () TLS/SSL configuration. Required for custom CA certificates or mTLS.
auth http:ClientAuthConfig () Authentication handler. Supports OAuth2, JWT, Basic, and Bearer token.
compression http:Compression AUTO Request/response compression. AUTO enables based on content type.
circuitBreaker http:CircuitBreakerConfig () Circuit breaker configuration. See section 3.3.
retryConfig http:RetryConfig () Retry policy configuration. See section 3.2.

Example with full configuration:

http:Client apiClient = check new ("https://api.example.com", {
    timeout: 30.0,
    httpVersion: http:HTTP_2_0,
    retryConfig: {
        count: 3,
        interval: 1.0,
        backOffFactor: 2.0,
        maxWaitInterval: 30.0
    },
    auth: {
        tokenUrl: "https://auth.example.com/oauth2/token",
        clientId: "my-client-id",
        clientSecret: "my-client-secret",
        scopes: ["read", "write"]
    }
});

3.2 Retry Configuration

The retry policy controls how the client handles transient failures. Retries apply only to idempotent requests (GET, PUT, DELETE) by default.

Field Type Default Description
count int 0 Maximum number of retry attempts (0 = no retries)
interval decimal 0.0 Initial delay between retries in seconds
backOffFactor float 0.0 Multiplier for exponential backoff (e.g., 2.0 = double delay each retry)
maxWaitInterval decimal 0.0 Maximum delay cap in seconds (prevents infinite backoff growth)
statusCodes int[] [503] HTTP status codes that trigger a retry
Warning Be careful when enabling retries for non-idempotent operations (POST). If the server processed the request but the response was lost due to a network issue, the retry will create a duplicate. Use idempotency keys or server-side deduplication when retrying POST requests.

3.3 Circuit Breaker

The circuit breaker prevents cascading failures by stopping requests to an unhealthy service. It operates in three states:

  1. Closed (normal operation) — requests flow through normally. Failures are counted.
    • When the failure rate exceeds the threshold, transitions to Open
    • Failure rate is calculated over a rolling time window
  2. Open (circuit tripped) — all requests immediately fail with a CircuitBreakerError
    • After the reset timeout, transitions to Half-Open
    • No actual HTTP calls are made — fails fast to protect the downstream service
  3. Half-Open (probe state) — a limited number of requests are allowed through
    • If they succeed, transitions back to Closed
    • If they fail, transitions back to Open
    • Only one probe request at a time by default
http:Client resilientClient = check new ("https://api.example.com", {
    circuitBreaker: {
        rollingWindow: {
            timeWindow: 60.0,       // 60-second rolling window
            bucketSize: 10.0,       // 10-second buckets
            requestVolumeThreshold: 20  // Minimum requests before tripping
        },
        failureThreshold: 0.5,     // 50% failure rate trips the circuit
        resetTime: 30.0,           // 30 seconds in Open state before probing
        statusCodes: [500, 502, 503]
    }
});

4. API Reference

4.1 HTTP Methods

The client supports all standard HTTP methods. Each method is available in two forms: resource method syntax and remote method syntax.

remote function get(string path, map<string|string[]>? headers = (), http:TargetType targetType = <>) returns targetType|http:ClientError
remote function post(string path, http:RequestMessage message, map<string|string[]>? headers = (), string? mediaType = (), http:TargetType targetType = <>) returns targetType|http:ClientError
remote function put(string path, http:RequestMessage message, map<string|string[]>? headers = (), string? mediaType = (), http:TargetType targetType = <>) returns targetType|http:ClientError
remote function delete(string path, http:RequestMessage message = (), map<string|string[]>? headers = (), string? mediaType = (), http:TargetType targetType = <>) returns targetType|http:ClientError
Deprecated remote function execute(string httpVerb, string path, http:RequestMessage message, ...) returns http:Response|http:ClientError
Note The execute method is deprecated since Swan Lake Update 10. Use the specific HTTP method functions instead, which provide better type safety and compile-time validation.

4.2 Response Handling

Responses can be bound to specific Ballerina types using the targetType parameter:

// Direct type binding — JSON deserialized to record
Album album = check client->/albums/["1"];

// Get raw response for status code inspection
http:Response response = check client->/albums/["1"];
int statusCode = response.statusCode;
Album album = check response.getJsonPayload().cloneWithType();

// String response
string html = check client->/pages/["home"];

// Binary response
byte[] pdf = check client->/reports/["monthly"];

5. Error Handling

The HTTP client uses a type hierarchy for errors, enabling fine-grained error handling:

Error Type Parent Description
http:ClientError error Base type for all client errors
http:ClientConnectorError ClientError Connection failures (DNS, TCP, TLS)
http:RemoteServerError ClientError Server returned 5xx status code
http:ClientRequestError ClientError Client-side errors (4xx status codes)
http:PayloadBindingError ClientError Response could not be deserialized to target type
http:CircuitBreakerError ClientError Circuit breaker is in Open state
http:AllRetryAttemptsFailed ClientError All retry attempts exhausted
http:IdleTimeoutError ClientError Connection idle timeout exceeded 
Album|error result = client->/albums/["1"];

if result is http:CircuitBreakerError {
    // Service is down, use cached data
    return getCachedAlbum("1");
} else if result is http:PayloadBindingError {
    // Response format changed, log and investigate
    log:printError("Album schema mismatch", result);
    return error("Invalid response from album service");
} else if result is http:ClientError {
    // All other client errors
    return error("Failed to fetch album", result);
} else {
    return result;  // Success — type is Album
}

6. Advanced Examples

6.1 Parallel API Calls with Workers

function fetchDashboardData(string userId) returns DashboardData|error {
    http:Client userApi = check new ("https://api.example.com/users");
    http:Client orderApi = check new ("https://api.example.com/orders");
    http:Client analyticsApi = check new ("https://api.example.com/analytics");

    // Parallel calls using workers
    worker userWorker returns UserProfile|error {
        return userApi->/[userId];
    }

    worker orderWorker returns Order[]|error {
        return orderApi->/users/[userId]/orders;
    }

    worker analyticsWorker returns Analytics|error {
        return analyticsApi->/users/[userId]/stats;
    }

    UserProfile profile = check wait userWorker;
    Order[] orders = check wait orderWorker;
    Analytics stats = check wait analyticsWorker;

    return {profile, orders, stats};
}

6.2 Streaming Large Responses

function downloadLargeFile(string fileId) returns error? {
    http:Client fileApi = check new ("https://files.example.com");

    http:Response response = check fileApi->/download/[fileId];

    // Stream response body to file
    stream<byte[], io:Error?> byteStream =
        check response.getByteStream();

    check io:fileWriteBlocksFromStream(
        "./downloads/" + fileId, byteStream
    );

    check byteStream.close();
}
